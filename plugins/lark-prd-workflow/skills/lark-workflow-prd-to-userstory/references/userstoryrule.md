# User Story 书写规则

## 概述

本文档定义 **PRD → User Story** 流程中的标准书写格式与规则，作为 **lark-workflow-prd-to-userstory** skill 的附件被加载使用。任何遵循"PRD 含 §4 功能清单"约定的项目（PST、CPQ、WLS 等）都可以直接复用本规则；不依赖任何特定产品的领域术语。

**核心约束：**

1. **数量对齐**：User Story 必须与对应 PRD 中 `# 4. 功能清单` 表格的功能项**一一对应**——功能清单有几条，对应的 P0 User Story 就有几条（一对一），不允许漏拆、合并或额外新增清单外功能。
2. **可验收**：每条 Story 必须配套 **可量化、可测试** 的验收标准（Acceptance Criteria）。
3. **优先级约束**：只生成 P0（最高优先级）User Story，与 PRD 功能清单优先级保持一致。

## 核心原则：INVEST

每个 User Story 必须满足 INVEST 原则：

### **I**ndependent（独立的）
- Story 应尽可能独立，避免相互依赖
- 可以独立开发、测试和部署
- 减少对其他 Story 的阻塞

### **N**egotiable（可协商的）
- Story 细节可以在实施前讨论和修改
- 允许开发团队与业务方协商实现方式
- 保持灵活性，避免过度详细的前期设计

### **V**aluable（有价值的）
- Story 应为目标用户（终端用户、业务方、内部运营、系统等）提供明确价值
- 能够解决具体的业务问题，价值可量化或被明确描述
- 验证「这条 Story 上线了，谁会因此受益、受益多少」

### **E**stimable（可估算的）
- 开发团队应能够估算完成 Story 所需的工作量
- 信息足够充分，可以进行合理估算
- 避免过于模糊或复杂的描述

### **S**mall（小型的）
- Story 应足够小，能在一个迭代内完成
- 建议 1-3 个开发日的工作量
- 如果过大，应拆分为多个 Sub-Story（但拆分后总数仍需对齐功能清单粒度）

### **T**estable（可测试的）
- Story 必须有明确的、可量化的验收标准
- 可以编写自动化或人工测试用例
- 测试结果能够明确判断是否完成

## 与 PRD 功能清单的对齐规则

### 1. 数量对齐

- 打开对应 PRD 文档，定位 `# 4. 功能清单` 章节
- **统计功能清单表格的有效行数 N**（排除表头与空行）
- 输出的 P0 User Story 数量必须**严格等于 N**
- 每条 Story 的标题或开头必须与功能清单中的"功能模块/功能描述"一一对应

### 2. 字段映射

| PRD 功能清单字段 | User Story 对应位置 |
|------------------|---------------------|
| 功能模块 | Story 标题（例如 `### User Story: <功能模块名>`） |
| 功能描述 | "我希望" + "以便" 段落，并展开为可执行描述 |
| 优先级 | Story 末尾的 `**优先级：** P0` |
| 备注（指向需求详细设计） | "背景"段落，引用 PRD 中对应的需求详细设计章节 |

### 3. 反向追溯

每条 Story 末尾必须标注 **PRD 来源**，格式：

```
**PRD 来源：** <PRD 路径> §<章节锚点>
```

示例：
```
**PRD 来源：** prd/2026/5/<feature>-prd.md §3. 需求详细设计 / <小节标题>
```

## 书写格式

### 标准结构

```
### User Story: <功能模块名，与 PRD 功能清单对齐>

**作为** <角色>
**我希望** <功能描述>
**以便于** <业务价值>

**背景：**
<引用 PRD 需求详细设计中的对应小节，简述业务问题>

**验收标准：**
- [ ] <触发条件> 时，系统 <执行动作>，并 <预期结果>
- [ ] 数值约束（数量上限、阈值、字段长度）必须写明具体数字
- [ ] 列举正常流程、异常流程、边界条件
- [ ] <可量化的额外验证点>

**技术需求：**（可选）
- <接口、数据结构、算法关键点>

**优先级：** P0（最高优先级）

**PRD 来源：** <PRD 路径与章节锚点>
```

### 角色（"作为 X"）选用建议

按需要选择最贴近场景的角色，避免空泛地写"用户"：

- **终端用户**：最终消费产品的人（消费者、玩家、读者等）
- **业务方**：使用产品支撑业务的人（销售、客服、运营、客户成功等）
- **内部运营/管理员**：管理后台或配置类用户（运维、管理员、审核员等）
- **集成方/合作伙伴**：通过 API 或工具集成产品的角色
- **系统**：当 Story 描述系统自动行为（如定时任务、自动同步）时

如果一个功能服务多个角色，按主要受益方写；次要角色写到「以便于」段落里。

### 验收标准书写规范

每条验收标准必须满足：

1. **三要素齐全**：触发条件 + 执行动作 + 预期结果
2. **拒绝模糊词**：禁用"适当"、"合理"、"友好"、"快速"、"流畅"、"完善"等主观描述
3. **数值化**：涉及阈值（尺寸、数量、时间、百分比、字段长度）必须给出具体数字
4. **覆盖边界**：必须包含至少 1 条异常 / 边界场景验收
5. **状态可观测**：验收点描述的结果必须能在 UI、日志、数据库或接口返回中被外部观察到

**反例 ❌：**
- [ ] 提示要友好
- [ ] 性能更好
- [ ] 加载快一点

**正例 ✅：**
- [ ] 当用户提交表单且必填字段为空时，对应字段下方在 200ms 内显示红色错误文案 "请填写 <字段名>"
- [ ] 列表页接口在 P95 响应时间 ≤ 800ms（由 APM 监控验证）
- [ ] 首屏加载在 4G 网络下 ≤ 2.5s（Lighthouse 测量）

## 拆分与边界

- 当某个功能清单项**实现复杂度过大**（>3 开发日），允许在该 Story 下挂载子 Story（Sub-Story），但 Sub-Story **不计入主 Story 数量**
- 主 Story 数量必须严格 = 功能清单条目数量
- 不允许把多个功能清单项合并为一个 Story
- 不允许凭空新增清单外的 Story

## 通用示例

假设某 PRD 功能清单包含 **3 条** 功能项 → 应输出 **3 条** P0 User Story。下面是其中一条的完整示例：

### User Story: 批量导出订单

**作为** 客户成功（CS）
**我希望** 在订单管理后台一次性勾选多条订单并导出为 Excel
**以便于** 月底向客户汇总账单时能一次拿到所有订单数据，减少手工拼装

**背景：**
当前订单后台只支持单条导出，CS 月底需对接 50+ 客户、每个客户 20–200 条订单，单条导出操作量大且易漏。详见 PRD §3. 需求详细设计 / 订单批量导出。

**验收标准：**

- [ ] 订单列表页表格首列展示复选框，支持单选、全选当前页、全选全部结果（≤ 10000 条）
- [ ] 用户勾选后，工具栏出现 `导出 Excel (<N>)` 按钮，N 为已勾选数
- [ ] 点击导出后，前端在 200ms 内跳出"正在生成导出文件"的提示；文件准备好后通过浏览器下载，文件名格式 `orders_YYYYMMDD_HHmmss.xlsx`
- [ ] 单次导出超过 10000 条时按钮置灰，hover 提示 "单次最多导出 10000 条订单，请缩小筛选范围"
- [ ] 导出的 Excel 第一行为表头，包含 12 个字段：订单号、客户名、状态、下单时间、…（顺序与 PRD §附录 A 一致）
- [ ] 导出请求在后端 P95 ≤ 5s（10000 条规模），超时返回 504 并在前端展示重试按钮
- [ ] 用户无 `order:export` 权限时按钮不展示，且后端接口返回 403

**技术需求：**
- 接口：`POST /api/orders/export`，请求体含订单 ID 数组与筛选条件
- 后端使用流式生成 xlsx，避免一次性加载所有数据到内存
- 审计日志记录：导出人、订单数量、导出时间

**优先级：** P0（最高优先级）

**PRD 来源：** `prd/2026/5/order-export-prd.md` §3. 需求详细设计 / 订单批量导出

## ❌ 反例（禁止出现）

```
### User Story: 体验优化大礼包
**作为** 用户
**我希望** 系统用起来更顺手
**以便于** 提升体验

**验收标准：**
- [ ] 提示更友好
- [ ] 性能更好
- [ ] 边界更合理

**优先级：** P2
```

问题：
1. 把多个功能清单项合并为一个 Story（违反一对一）
2. 验收标准模糊不可测（违反"拒绝模糊词"和"三要素齐全"）
3. 优先级非 P0
4. 角色写成空泛的"用户"，未明确受益方

## 提交前检查清单

提交 User Story 之前，必须逐项检查：

- [ ] **Story 数量 = PRD 功能清单条目数量**
- [ ] 每条 Story 都对应到功能清单的某一行（一对一映射）
- [ ] 每条 Story 优先级均为 P0
- [ ] 每条 Story 满足 INVEST 原则
- [ ] 角色（"作为 X"）已选取最贴近的具体角色，未空泛写"用户"
- [ ] 每条验收标准三要素齐全（触发条件 + 执行动作 + 预期结果）
- [ ] 验收标准包含数值约束与边界 / 异常场景
- [ ] 没有主观模糊词（"适当"、"合理"、"友好"、"流畅"等）
- [ ] 标注了 PRD 来源章节
- [ ] 单条 Story 工作量在 1-3 个开发日内（超出则拆 Sub-Story）

## 与 prd-to-userstory skill 的协作约定

本文档作为 **lark-workflow-prd-to-userstory** skill 的规则附件被加载，skill 在拆解 PRD 时必须：

1. 先读取目标 PRD 文档，定位 `# 4. 功能清单` 表格
2. 加载本规则文件
3. 按 "数量对齐 → 字段映射 → 验收标准" 顺序逐条产出 Story
4. 输出后自动执行"提交前检查清单"，发现违规则重新生成

## 项目特定补充

如某个项目（如 PST、CPQ、WLS）需要在通用规则之外补充自身术语、角色、字段映射，请在 **同级 references 目录新增 `userstoryrule-<project>.md`** 形式的子规则文件，由 SKILL.md 按项目分支加载，**不要修改本通用规则文件**。

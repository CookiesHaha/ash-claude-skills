# PRD 编写规范

本规范定义 PRD（产品需求文档）的写作约束，确保需求清晰、可执行，并便于 AI 辅助理解与实现。
被 [`SKILL.md`](SKILL.md) 在 Step 3 生成 PRD 时强制约束，也可独立作为 PRD 编写者的写作规范使用。

---

## 规范一：用结构化语言描述需求，避免模糊表述

每条需求必须包含**触发条件、执行动作、预期结果**三要素，禁止使用"适当"、"合理"、"友好"等主观模糊词。

**不规范示例：**

> 页面加载要快，用户体验要好。

**规范示例：**

> 当用户进入订单列表页时，列表数据应在 500ms 内完成首屏渲染；若超时则展示骨架屏，并在 3s 内完成数据加载，否则显示"加载失败，点击重试"提示。

---

## 规范二：以数据和状态驱动需求，明确所有边界场景

需求描述必须列举**正常流程、异常流程、边界条件**，并对每种状态给出明确的系统行为定义。数值约束（数量上限、时间阈值、字段长度等）必须写明具体数字。

**不规范示例：**

> 表单支持添加多个项目，过多时给出提示。

**规范示例：**

> 每个表单最多支持添加 8 个项目。
>
> - 当已添加项目数 < 8 时，"添加项目"按钮可点击。
> - 当已添加项目数 = 8 时，"添加项目"按钮置灰，hover 提示"最多支持 8 个项目"。
> - 删除项目后，按钮恢复可点击状态。

---

## 规范三：为每条需求附加上下文背景与验收标准

每条需求应说明**业务背景**（为什么做）和**验收标准**（怎样算完成），使 AI 和开发者无需猜测意图即可直接实现和验证。

**格式模板：**

```
### 需求名称

**背景：** [说明该需求解决什么问题，或来自哪个业务场景]

**需求描述：**
[用结构化语言描述功能逻辑]

**验收标准：**
- [ ] [可量化的验收条件 1]
- [ ] [可量化的验收条件 2]
- [ ] [异常场景验收条件]
```

**示例：**

### 订单批量导出

**背景：** CS 月底向客户汇总账单时需对接 50+ 客户、每个客户 20–200 条订单，原系统只支持单条导出，操作量大且易漏。

**需求描述：**
订单列表页支持勾选多条订单后一次性导出为 Excel；单次导出上限 10000 条；超过则按钮置灰并提示。

**验收标准：**

- [ ] 列表页表格首列展示复选框，支持单选 / 全选当前页 / 全选全部结果（≤ 10000 条）
- [ ] 用户勾选后，工具栏出现 `导出 Excel (<N>)` 按钮，N 为已勾选数
- [ ] 点击导出后，前端在 200ms 内跳出"正在生成导出文件"提示；文件准备好后通过浏览器下载，文件名格式 `orders_YYYYMMDD_HHmmss.xlsx`
- [ ] 单次导出超过 10000 条时按钮置灰，hover 提示"单次最多导出 10000 条订单，请缩小筛选范围"
- [ ] 用户无 `order:export` 权限时按钮不展示，且后端接口返回 403
